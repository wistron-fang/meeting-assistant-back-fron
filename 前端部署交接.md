## 0. 现状速读

- **技术栈**：Vite 6 + React 19 + antd 5 + axios + react-router-dom 6 + valtio。是个 SPA（单页应用）。
- **构建**：`npm run build` → 默认输出 `frontend/dist/`（纯静态文件，交给 nginx 托管即可）。
- **API 地址来源**：axios 的 `baseURL = import.meta.env.VITE_API_BASE`（见 `src/api/request/index.ts`）。
- **⚠️ 最关键的一点**：`VITE_API_BASE` 是 **Vite 环境变量，构建时被烤进 bundle**。也就是说——**改完 `.env` 必须重新 `npm run build`** 才生效，不能构建后再改。
- **当前 `frontend/.env`**（开发态，未改）：

  VITE_API_BASE = http://localhost:8100/      ← 上线必须改这个

- **后端 API**：跑在 ECS 的 `:8100`，路由前缀是 `/auth`、`/meeting`、`/hello`（**没有 `/api` 前缀**）。
- **后端 CORS**：`app_main.py` 当前 `allow_origins=["*"]`（全放开），下载文件名靠 `expose_headers=["Content-Disposition"]`。

1. **用户怎么访问？** 公网 IP（如 `http://47.x.x.x`）还是 **域名**（如 `meeting.gigba-ai.cn`）？ → **选域名**

## 2. 两条部署路径（按决策二选一）

### 方案 A：纯 IP / HTTP（最快，无域名无证书，适合先跑通）
- nginx 在 80 端口托管 `dist/`；API 仍在 `:8100`，安全组放通 **80 + 8100**。
- `frontend/.env` 设 `VITE_API_BASE = http://<ECS公网IP>:8100/`。
- 前后端都是 http，无 mixed content 问题。CORS 已是 `*`，能通。
- 缺点：地址丑、无加密、API 端口直接暴露。仅建议作过渡。

### 方案 B：域名 + HTTPS + nginx 反代（推荐，给真实用户）
- 域名解析到 ECS 公网 IP；nginx 在 **443** 托管 `dist/` 并**反向代理 API**：
  - `/`               → 静态文件 `dist/`
  - `/auth/` `/meeting/` `/hello` → `http://127.0.0.1:8100`
- 这样前端 axios 走**同源 https**，`VITE_API_BASE = /`（相对路径），**绕开 CORS + mixed content**。
- 证书用 Let's Encrypt（`certbot`），香港免备案可直接签。
- 安全组只需放通 **80 + 443**（8100 不必对外）。

> 推荐 **方案 B**：同源反代最干净，不用碰 CORS，API 端口也不暴露。下面步骤以 B 为主，A 作简化。

  - 验证解析生效：`dig +short meeting.gigba-ai.cn`（或 `nslookup meeting.gigba-ai.cn`）应返回 `47.76.244.232`。
- ECS 安全组放通 **80、443**（8100 不必对外）。

### 3.2 改前端 .env 并构建（本地或 ECS 上都行）
```bash
# frontend/.env 改成同源相对路径（配合 nginx 反代）
VITE_TITLE = 会议纪要助手
VITE_API_BASE = /
```
```bash
cd frontend
npm install          # 首次
npm run build        # 产出 frontend/dist/
```
> ⚠️ 确认 `src/api/request/index.ts` 的 baseURL 用的是 `VITE_API_BASE`；若代码里 API 路径已自带 `/auth`、`/meeting`，则 `VITE_API_BASE=/` 后请求会落到 `https://域名/auth/...`，正好被 nginx 反代接走。

### 3.3 把 dist 传到 ECS
- Workbench 上传 `dist.tar.gz` → 解压到如 `/opt/meeting/frontend/dist`。

### 3.4 装并配 nginx
```bash
yum install -y nginx || dnf install -y nginx
systemctl enable --now nginx
```
nginx 站点配置（**已生成好可直接用**：`deploy/nginx/meeting.conf`，上传为 `/etc/nginx/conf.d/meeting.conf`；先 80，证书装好后 certbot 会自动加 443）：
```nginx
server {
    listen 80;
    server_name meeting.gigba-ai.cn;

    root /opt/meeting/frontend/dist;
    index index.html;

    # SPA 路由兜底：react-router 的前端路由刷新不 404
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API 反向代理到后端（注意保留 Content-Disposition 给下载文件名）
    location ~ ^/(auth|meeting|hello) {
        proxy_pass http://127.0.0.1:8100;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        client_max_body_size 200m;          # 音频/文档上传，别被默认 1m 卡死
        proxy_read_timeout 300s;            # 长任务接口留足
    }
}
```
```bash
nginx -t && systemctl reload nginx
```

### 3.5 上 HTTPS（Let's Encrypt）
```bash
yum install -y certbot python3-certbot-nginx || dnf install -y certbot python3-certbot-nginx
certbot --nginx -d meeting.gigba-ai.cn        # 自动改 nginx 配置加 443 + 跳转
systemctl reload nginx
```

### 3.6 验证
- 浏览器开 `https://meeting.gigba-ai.cn` → 能加载界面、能登录（用 `admin_create_user.py` 建的号）、能提交任务、能下载产物。
- 刷新子路由页面不 404（SPA 兜底生效）。
- 下载文件名正常（Content-Disposition 透传）。

## 4. 方案 A（纯 IP/HTTP）的差异
- `frontend/.env`：`VITE_API_BASE = http://<ECS公网IP>:8100/` → `npm run build`。
- nginx 只托管静态（同上 `try_files` 兜底），**不需要反代**（前端直连 8100）。
- 安全组放通 80 + 8100。
- 跳过 3.5 证书步骤。

---

## 5. 易踩的坑

1. **`VITE_API_BASE` 构建时固化**：改了 `.env` 必须重新 `npm run build`，否则 dist 里还是旧地址。
2. **SPA 路由 404**：nginx 必须 `try_files $uri $uri/ /index.html`，否则刷新 `/xxx` 子页面报 404。
3. **mixed content**：https 前端 + http API 会被浏览器拦。方案 B 同源反代天然规避；方案 A 全 http 也没事；最忌讳"前端 https、API 写成 http://IP:8100"。
4. **上传体积**：nginx 默认 `client_max_body_size 1m`，音频/文档上传会 413，必须调大（示例给了 200m）。
5. **下载文件名**：反代要透传响应头，后端已 `expose_headers=Content-Disposition`，nginx 默认会传，别手动过滤掉。
6. **CORS（仅方案 A 相关）**：后端现 `allow_origins=["*"]` 能用；但 `allow_origins=* + allow_credentials=True` 组合在带 cookie 时浏览器会拒——本项目用 JWT 放 header（非 cookie），所以目前没事。若以后改 cookie 鉴权需收紧 origin。方案 B 同源则完全无 CORS。
7. **长任务超时**：提交/轮询接口走反代，`proxy_read_timeout` 留足（示例 300s）。
