python app_main.py 只启动了 FastAPI（接 HTTP 请求），并不会处理后台任务。真正干活的是 Celery worker，它是另一个独立进程，要单独跑。

解决：再开一个窗口启动 Celery worker
新开一个 PowerShell 窗口，按下面操作（不要关闭已经在跑的后端窗口和前端窗口）：

cd E:\meeting_minutes_assistant\backend\app
.\.venv\Scripts\Activate.ps1
celery -A core.celery_app worker --loglevel=info --pool=solo

--------------------------
service 模块找不到——celery_app.py 里虽然有把 backend/app 加到 sys.path 的代码，但在 worker 启动的某个阶段没生效。最直接的修复：启动前显式设置 PYTHONPATH。

在那个 PowerShell 窗口里执行：

$env:PYTHONPATH = "E:\meeting_minutes_assistant\backend\app"
celery -A core.celery_app worker --loglevel=info --pool=solo
$env:PYTHONPATH 是 PowerShell 设置环境变量的语法，告诉 Python 在导入模块时也去这个目录找。这样 service.meeting 就能被找到了。

注意$env:PYTHONPATH 只在当前 PowerShell 窗口生效，关了就没了。所以以后每次新开窗口启动 worker 都要先跑一次这条 export 命令。

如果你嫌每次手动设烦，可以写个一键启动脚本（以后再说，先把任务跑通要紧）。
