# 前端部署文档（会议纪要助手）

> 自包含操作手册，供后续任何人对照部署 / 更新前端。
> 已按本文于 2026-06-04 实际上线：**https://meeting.gigba-ai.cn**。
> 配套文件：`deploy/nginx/meeting.conf`（nginx 配置）、`deploy/dist.tar.gz`（最近一次构建产物）。

---

## 0. 架构与关键事实（先读这段，能少踩一半坑）

- **技术栈**：Vite 6 + React 19 + antd 5 + react-router-dom 6（SPA 单页应用）。
- **部署方案 = 方案 B：域名 + HTTPS + nginx 同源反代**。一台 nginx 同时：
  - `/` → 托管静态文件 `dist/`
  - `/auth/` `/meeting/` `/hello` → 反向代理到后端 `http://127.0.0.1:8100`
  - 前端 axios 走**同源 https**，因此 `VITE_API_BASE=/`（相对路径），**天然零 CORS、零 mixed content**，后端 8100 端口无需对外。
- **⚠️ 头号坑：`VITE_API_BASE` 是 Vite 构建期变量，会被烤进 bundle**。改 `.env` 后**必须重新 `npm run build`** 才生效；不能 build 完再改 .env。
- **后端 API 路由前缀**：`/auth`、`/meeting`、`/hello`（**没有 `/api` 前缀**）。nginx 反代的 location 必须与此一致。
- **前端只能本地构建后上传**：服务器上是压缩混淆后的产物，且后端机未装 node，**无法在服务器上改前端或构建**。前端每次改动 = 本地改源码 → build → 上传 dist。（后端 .py 才能在服务器直接改+重启。）

## 1. 前置条件（用户侧，首次部署必须先完成）

1. **域名实名认证**：`.cn` 域名未实名则根本不解析。阿里云域名控制台（dc.console.aliyun.com）→ 域名列表 → 看「实名认证」列为已认证。
2. **加 A 记录**：阿里云云解析（dns.console.aliyun.com）→ 对应域名 → 添加记录：
   - 记录类型 `A`、主机记录 `meeting`（即子域名）、记录值 = ECS 公网 IP、TTL 600。
   - 验证：`nslookup meeting.gigba-ai.cn 223.5.5.5` 返回该 IP（解析传播通常几分钟，certbot 前务必先生效）。
3. **ECS 安全组**：入方向放通 **80、443**（源 `0.0.0.0/0`）。8100 无需对外。

## 2. 本地：构建并打包（在开发机 frontend/ 目录）

```bash
# 2.1 确认 .env 为同源相对路径（生产配置）
#   frontend/.env:
#     VITE_TITLE = 会议纪要助手
#     VITE_API_BASE = /          ← 关键，配合 nginx 反代
#     VITE_API_PROXY = http://localhost:8100/   ← 仅 dev proxy 用，生产无关

cd frontend
npm install            # 首次
npm run build          # 产出 frontend/dist/

# 2.2 打包成可上传的压缩包（输出到 deploy/dist.tar.gz）
tar -czf ../deploy/dist.tar.gz -C dist .
```

> 校验：`tar -tzf ../deploy/dist.tar.gz` 应看到 `./index.html`、`./assets/...`、`./gba-logo.png`（favicon）。

## 3. 服务器：首次部署（在 ECS，阿里云 Workbench，root）

> 解析未生效前可先做 3.1~3.2（HTTP 80 跑通），解析 OK 后再做 3.3（HTTPS）。

### 3.1 上传 + 解压静态包
- Workbench「上传文件」把 `deploy/dist.tar.gz` 和 `deploy/nginx/meeting.conf` 传到 `/root/`。
```bash
mkdir -p /opt/meeting/frontend/dist
tar xzf /root/dist.tar.gz -C /opt/meeting/frontend/dist
ls /opt/meeting/frontend/dist     # 应见 index.html assets/ gba-logo.png
```

### 3.2 装并配 nginx（先 80）
```bash
yum install -y nginx || dnf install -y nginx
cp /root/meeting.conf /etc/nginx/conf.d/meeting.conf
nginx -t                          # 必须 syntax is ok / test is successful
systemctl enable --now nginx
systemctl reload nginx

# 自测
curl -I http://127.0.0.1/         # 期望 200（静态 index.html）
curl http://127.0.0.1/hello       # 期望后端 JSON（验证反代通了）
curl -I http://127.0.0.1:8100/hello   # server: uvicorn 即后端在跑（HEAD 返回 405 是正常的）
```

### 3.3 上 HTTPS（解析生效后）
```bash
yum install -y certbot python3-certbot-nginx || dnf install -y certbot python3-certbot-nginx
certbot --nginx -d meeting.gigba-ai.cn
#   邮箱：填真实邮箱（收到期提醒）
#   条款：A 同意；EFF 订阅：N
#   redirect HTTP→HTTPS：选 2（强制跳转）
systemctl reload nginx
```
certbot 会自动改 `/etc/nginx/conf.d/meeting.conf` 加 `listen 443 ssl` + 证书 + 80→443 跳转，并设好自动续期。

### 3.4 验证（公网）
```bash
curl -I https://meeting.gigba-ai.cn/      # 200
curl -I http://meeting.gigba-ai.cn/       # 301 跳 https
curl    https://meeting.gigba-ai.cn/hello # 后端 JSON
```
浏览器开 https://meeting.gigba-ai.cn：能登录、提交任务、下载产物；刷新子路由不 404；上传大文件不 413。

## 4. 日常更新前端（最常用，改完页面走这个）

```bash
# 本地
cd frontend
npm run build
tar -czf ../deploy/dist.tar.gz -C dist .

# 上传新 deploy/dist.tar.gz 到 ECS /root/ 后，在 ECS 执行：
rm -rf /opt/meeting/frontend/dist/*
tar xzf /root/dist.tar.gz -C /opt/meeting/frontend/dist
# 纯静态更新，无需 reload nginx（仅当改了 nginx 配置才需 nginx -t && systemctl reload nginx）

# 浏览器 Ctrl+Shift+R 强刷（绕过缓存）
```

## 5. 用户管理（后端，非前端但常一起做）

自助注册已关闭，账号由管理员建：
```bash
cd /opt/meeting/backend/app
# 单个：
/root/miniconda3/envs/meeting/bin/python admin_create_user.py <用户名> <邮箱> <密码> [--star 4] [--superuser]
# 批量：编辑 batch_create_users.py 里的 USERS 列表后运行（生成 created_users.csv，下载后删除）：
/root/miniconda3/envs/meeting/bin/python batch_create_users.py
```
> ⚠️ 邮箱必须是**合法格式**（如 `name@gigba.org.cn`）。`UserResponse.email` 是 `EmailStr`，若邮箱用了 `.local` 等保留域，登录会在序列化响应时 500（见第 6 节坑 4）。

## 6. 易踩的坑（本次实战记录）

1. **`VITE_API_BASE` 构建期固化**：改 .env 必须重新 build，否则 dist 里还是旧地址。
2. **SPA 路由 404**：nginx 必须 `try_files $uri $uri/ /index.html`，否则刷新子页面 404。
3. **certbot 首签报 `CAA SERVFAIL during secondary validation`**：新建 A 记录传播未稳的临时问题。**等几分钟原样重跑 `certbot --nginx -d 域名` 即成功**，不必加 CAA 记录（账号已注册会直接进签发）。
4. **登录 500（不是 401）**：管理员/用户邮箱非法（如 `xxx@admin.local`）→ 登录认证通过但返回 `UserResponse` 时 `EmailStr` 校验抛异常 → 500。修法：把该用户邮箱改成合法值。看真实报错：`journalctl -u meeting-api -n 60 --no-pager`。
5. **登录后跳 404**：`login.tsx` 里登录成功的默认跳转目标必须是真实存在的路由（本项目是 `/`；曾误写 `/chat` 导致 404）。
6. **上传体积 413**：nginx 默认 `client_max_body_size 1m`，音频/文档上传会失败，配置里已设 `200m`。
7. **长任务超时**：反代 `proxy_read_timeout` 要够大（配置里 300s）。
8. **下载文件名**：后端已 `expose_headers=Content-Disposition`，nginx 默认透传，别手动过滤。
9. **本地预览 dist 时白屏/500**：`vite preview` 会继承 `vite.config.ts` 的 `server.proxy`，而生产 `.env` 的 `VITE_API_BASE=/` 让 proxy 把所有请求转发到没运行的 8100 → 500。预览构建产物请改用不带 proxy 的静态服务器：`npx serve -s frontend/dist -l 4173`。
