> 写作日期：2026-05-30。读完「0 必读」「1 架构」两节就能开始干活，第 4 节是四块工作的落地细节。

## 0. 必读：先理解并发模型（这决定了一切）

**「100 用户并发」≠「100 任务并发」。** 本系统每个任务都是重活、长耗时：

- 音频先走阿里云 ASR（长录音几分钟）
- 再走 LLM 流水线（`minutes_engine` 7 步、多次大模型调用、langgraph 状态机，吃内存），单任务超时设了 **40 分钟**（`core/celery_app.py` `task_time_limit=60*40`）
- 最后渲染 PDF / DOCX

所以分两层看，瓶颈完全不同：

| 层 | 100 并发能不能扛 | 真正的瓶颈 | 对策 |
|---|---|---|---|
| **HTTP 层**（登录 / 提交 / 轮询状态） | ✅ 轻松 | DB 连接池太小（默认 5+10） | gunicorn 多 worker + 调大连接池 |
| **任务执行层**（真正跑纪要） | ❌ 100 个不可能同时跑 | LLM API 限流 + worker 内存/CPU；单机现实并发 **4~16** | 排队 + 横向扩 worker + 星级优先 |

**核心结论**：100 人同时在线、提交、看进度 → 没问题。100 个任务想同时跑完 → 不现实，它们必然**排队、分批消化**。
"按星级排队"正是为这个排队场景设计的——**产能不够时，决定谁的任务先被 worker 取走**。方向完全正确。

## 0.5 生产实测约束与容量测算（基于真实 API 配额 + ECS/PG 配置）

> 2026-05-31 拿到生产环境的 DashScope 限流配额 + 服务器配置，把"现实并发上限"从估算变成可算。

### 已知硬约束

| 资源 | 数值 | 是否瓶颈 | 说明 |
|---|---|---|---|
| deepseek-v4-pro RPM | 10,000/分（166/秒） | ❌ | 调用次数远够 |
| **deepseek-v4-pro TPM** | **1,200,000/分**（含输入+输出） | ✅ **LLM 侧天花板** | 决定 LLM 并发上限 |
| text-embedding-v3 | RPM 6,000 / TPM 24,000,000 | ❌ | RAG 嵌入用，富余 20 倍 |
| ECS `ecs.e-c1m2.xlarge` | **4 vCPU / 8 GiB** | ✅ **单机硬限** | langgraph 重，每并发任务约 0.5–1.5G 内存 |
| PostgreSQL 15 | 2048M 内存 / 最大 IOPS 3100 | ❌ | 元数据够用，注意连接数 + 轮询 |
| 阿里云 ASR (NLS/OSS) | 表中未给，**需单独确认** | ⚠️ | 音频任务长尾，100 人传音频要另算 OSS 带宽+ASR 并发 |

### 单任务 token 测算（基于真实输入：原文 3–5 万字）

| 环节 | 估算 |
|---|---|
| 原文字数 | 30,000–50,000 字 |
| → 输入 token | ~20k–50k（DeepSeek/Qwen 中文 ≈ 0.6–1.0 token/字） |
| 流水线读全文遍数 | 分块摘要 + 主题提取 + 分章节提取，约 3–6 遍 |
| 输出 token | ~30k–50k |
| **单任务总 token X** | **≈ 120k–280k，按 ~180k 估** |

### 单机并发上限测算

公式：TPM 允许并发 ≈ 1,200,000 ÷ (X ÷ LLM活跃分钟)；内存允许并发 ≈ 8G ÷ 单任务内存。取小。

| 约束 | 在 X≈180k 下 | 结论 |
|---|---|---|
| DeepSeek TPM 120万/分 | 单任务 ~30k/分 → 允许 **~25–40 并发** | ❌ **非瓶颈，富余 4–7 倍** |
| ECS 8G 内存 | 每任务 ~0.5–1G（langgraph 占用为主，文本才几 MB） | ✅ **唯一的墙，~6–8 并发** |

**结论：这台 4C8G 单机现实并发 ~6–8 个任务，规划按 6 算，被内存卡死，TPM 大量富余。**
结论很稳：TPM 要成瓶颈得单任务超 **~90 万 token**，3-5万字差 3–5 倍够不着——所以不管流水线读几遍，内存都是先到顶的。

### 排空时间（100 突发提交，单任务 ~5–8min 偏长）

| | 单机 ~6 并发 | 4 台 ~24 并发 |
|---|---|---|
| 100 突发排空 | **~1.5–2 小时** | **~25 分钟** |

→ 突发场景单机排队体验差，**4 台 + 星级优先**才能让高星用户几分钟内跑上。音频任务更久。

### 扩容杠杆（已按真实字数修正：TPM 有余量，横向扩有效）

- **横向（加 worker 机）**：**最实用**。TPM 富余，可扩到 **~4 台（~24–32 并发）才逼近 TPM 墙**（落地改造见工作五）。
- **纵向（更大内存机）**：单机并发可上 10–15，仍受同一 TPM 限制，但因 TPM 有余量、收益比之前判断好。
- **① 提高 DeepSeek TPM 配额**：扩过 ~4 台后的下一步。
- **② 降单任务 token**：流水线少读几遍 / 子步骤换便宜模型（`config/llm_config.py` 已对部分 Agent 用 qwen-plus）。

### ⚠️ TPS 子限制（隐藏坑）

限流表注明"按 TPS(=TPM/60) 限制" = **2 万 tokens/秒**。若流水线某步**单次发整篇 3-5万字（2-5万 token）**，单请求就逼近秒级上限，两任务同发即 429——哪怕分钟级 TPM 还空。**对策**：大文本必须分块调用（"摘要第 X/N 块"方向正确），杜绝"单次发全文"。压测重点：**TPM 有余量却仍 429 = 撞了 TPS**。

### 部署拓扑建议（针对这套配置）

1. 这台 4C8G **专给 Celery worker**，`-c 4~6`（压测定准）。
2. Web 层 gunicorn 只起 2–3 worker（HTTP 很轻）；预算够就把 web+nginx+redis+beat 拆到独立小机器。
3. **⚠️ 别贸然换线程池省内存**：`process_meeting` 用进程级 `redirect_stdout` 抓进度（`tasks.py:181`），线程池下会串台。想用 `-P threads/gevent`（适合这种等远程 API 的 I/O 型负载、省内存）必须**先重构进度抓取**。
4. PG 连接池放小（每 web worker ~10），用 Redis 缓存任务状态减轻 100 人轮询。2G/3100 IOPS 跑元数据绰绰有余。

> 一句话给下个 LLM（⚠️ 已被 2026-05-31 实测修正，以 0.6 节为准）：**token 便宜、TPM 完全不是瓶颈（要 ~250 并发才碰）；真瓶颈是「任务慢」（~16min/2.5万字，全是串行 LLM 调用）。横向扩可随便加机器；最高性价比优化是把流水线串行调用并行化。**

## 0.6 实测结果 + 部署进度（2026-05-31 跑通，2026-06-01 完成多样本测试）

> 这一节是真实数据，**推翻了 0.5 节的估算**（0.5 估算表保留作对照，以本节为准）。

### 多样本实测（4 个干净数据点）

| task | 输入字数 | 块数 | 调用 | 总token | 耗时 | token/分 |
|---|---|---|---|---|---|---|
| 1（单跑） | 25,585 | 11 | 31 | 73,807 | 15.8min | 4,672 |
| 2（4并发） | 25,585 | 11 | 26 | 67,624 | 16.1min | 4,200 |
| 3 | 35,224 | 6 | 19 | 60,061 | 11.8min | 5,090 |
| 4 | 53,943 | 11 | 24 | 83,270 | 18.4min | 4,525 |
| **均值** | — | — | **25** | **~71k** | **15.5min** | **~4,620** |

模型全 deepseek-v4-pro；embedding 用 text-embedding-v3（按段落数分批，2.5万字=44批，全串行 ~19s）。

### 核心结论（已实测确认）

- **token/时间不随「字数」线性，随「块数/调用数」走**：task3（3.5万字/6块）比 task2（2.5万字/11块）还便宜还快。块数由内容语义结构定。→ **配额按均值+buffer，别用字数公式。** 另叠 ~10% 跑次方差（同文件 73,807 vs 67,624）。
- **token 速率恒定 ~4,620/分钟**（4,200-5,090），与文档大小无关（token 和时间一起涨）。这是定并发的锚。
- **并发开销可忽略**：4 并发 task2(965s) vs 单跑 task1(946s) 只 +2% → I/O 型确认。
- **内存实测不是瓶颈**：4 并发 = 空闲基线（~1.26G used / 6.2G 空闲，COW 共享 langchain）。单机 -c 可推很高。
- **没撞任何限流**（grep 429 无输出）。

### 并发上限（实测数据算）

| 约束 | 能扛并发 | 说明 |
|---|---|---|
| **DeepSeek TPM** 120万 | **~260** | 稳态主天花板，与文档大小无关 |
| DeepSeek RPM 1万 | ~6,000 | 不限 |
| **Embedding RPS** 100/秒 | **~40**（突发启动）| 每任务开头猛打几十次 embedding；大文档更易撞，但有 429 重试退避，自动放慢不失败 |
| ECS 内存 | 实测不限 | — |

### 🎯 吞吐结论（修正后更乐观）

内存不是瓶颈 → 单机 `-c` 可推 16-32：

| 单机 -c | 吞吐 | 100 突发排空 |
|---|---|---|
| 6（旧悲观估）| 0.4/min | ~4 小时 |
| 16 | 1.0/min | ~1.6 小时 |
| **32** | 2.1/min | **~1 小时** |

**单台 4C8G 推到 -c 32，100 突发约 1 小时排空。** 加机器线性提升，API 额度够 ~8-15 台。

### 真瓶颈 & 杠杆（最终定性）

- **不是** TPM（260 余量）、**不是**内存（平的）。
- **是**单任务慢（15.5min，串行调用）+ 单机 -c 实际上限。
- **杠杆**：① **把 -c 推到 32**（零成本，内存白送）最快的赢；② 流水线并行砍到 ~5min；③ 加机器。

### 星级配额建议（单任务按 ~90k token = 均值71k×1.25buffer）

| 星级 | 月任务数 | 月token配额 | 并发 | 优先级 |
|---|---|---|---|---|
| ⭐⭐⭐⭐⭐ | 200 | 1800万 | 5 | 最高 |
| ⭐⭐⭐⭐ | 100 | 900万 | 4 | 高 |
| ⭐⭐⭐ | 50 | 450万 | 3 | 中 |
| ⭐⭐ | 20 | 180万 | 2 | 低 |
| ⭐ | 10 | 90万 | 1 | 最低 |

### -c 32 并发压测结果（2026-06-01，32 个真实 .docx 一秒内全提交）

| 指标 | 结果 |
|---|---|
| 内存峰值 | 4167M used / **3295M avail**（用 56%，剩 3.3G）|
| CPU load | 全程 **~0.00**（4 核几乎全闲）|
| 失败 | **0** |
| 完成 | ~20-25min 全完，最长单任务 24.5min（大文档）|
| 限流 429 | 341 次，**embedding 335 / DeepSeek 1** |

**瓶颈是 embedding RPS(100/秒)，不是内存/CPU/DeepSeek。** 335 个 embedding 429 全被自动重试救回(0失败)，是"32个同时提交"的最坏情况，真实错峰会少很多。DeepSeek 几乎没碰(1个429)，~260 并发上限稳。

### 🏁 最终部署结论（重大简化）

- **单台 4C8G 跑 `-c 32` 即可撑"100 并发用户"，多机非必需**（之前 8-15 台规划降级为按需）。100用户≠100同时任务；32同时也~20min清空。
- box 在 -c 32 下严重空闲(CPU 0.0/剩3.3G)，还能更高，但 **-c 越高 embedding 突发429越多**，收益递减。
- 生产参数：单 worker **`-c 32`**；`MAX_CONCURRENT_PER_USER` 按星级；多机先不上。

### 头号性价比优化：并行 DeepSeek chat 调用（注意：≠ embedding）

- DeepSeek 余量巨大(32并发才1个429) → **并行单任务内独立 chat 调用安全**（11块摘要+11主题），单任务 **~15min → ~5min**。
- ⚠️ **embedding 不能再并行**（已是瓶颈，并行=火上浇油）；只并行 **DeepSeek chat**。
- 改的坑：①token计量 contextvar 不跨线程(漏算，要 copy_context) ②进度抓取 redirect_stdout 非线程安全 ③高并发聚合可能蹭 DeepSeek TPS(2万/秒)，重试可扛。
- 上线后的头号优化，单独做，别现在动引擎。

### 工作二：Token 计量 + 配额

**目标**：统计每个任务消耗的 token，累加到用户月度用量；超月度预算则拒绝新任务。
**这是四块里唯一要动核心 LLM 引擎的，工作量最大。**

**关键事实**：所有 LLM 调用都过**唯一入口** `LLMClient.invoke_safe`（`minutes_engine.py:343`），
内部 `response = self.llm.invoke(messages)`（`:352`）。langchain-openai 0.1.x 下，
`response.response_metadata["token_usage"]` 含 `{prompt_tokens, completion_tokens, total_tokens}`
（DashScope 兼容模式会返回 usage）。已核实全文 ~20 处 LLM 调用全走 invoke_safe；Embedding 走独立
`EmbeddingClient`（text-embedding-v3，另算，TPM 富余 20 倍），本计量暂不含。

> **进度（2026-05-31）**：计量探针**已实现**（压测读数版，仅打日志、未写库）：
> - 新增 `service/meeting/usage_meter.py`（contextvar 累加器）
> - `minutes_engine.py:352` invoke_safe 内挂钩 `add_usage`（token_usage / usage_metadata 双兜底）
> - `tasks.py` `process_meeting` 开头 `start_metering()`、generate_minutes 后打 `[TOKEN计量]` 日志
> - **剩余（本工作二未完成部分）**：建 `task_usage` 表 + `users` 配额列、collect() 写库、累加
>   `tokens_used_this_month` + 月度重置、create_tasks 配额拦截。压测拿到真实数字后再做。

**步骤**：

1. **加累加器（contextvar）**：新建 `service/meeting/usage_meter.py`：
   ```python
   import contextvars
   _usage = contextvars.ContextVar("llm_usage", default=None)

   def start_metering():
       _usage.set({"prompt": 0, "completion": 0, "total": 0})

   def add_usage(prompt, completion, total):
       cur = _usage.get()
       if cur is not None:
           cur["prompt"] += prompt; cur["completion"] += completion; cur["total"] += total

   def collect():
       return _usage.get() or {"prompt": 0, "completion": 0, "total": 0}
   ```
   > 用 contextvar 而非全局变量：prefork worker 一进程一任务时全局也行，但 contextvar 对
   > `-P threads`（Windows dev）和异步同样安全，一步到位。
2. **在唯一入口挂钩**：`minutes_engine.py:352` 拿到 `response` 后：
   ```python
   try:
       u = response.response_metadata.get("token_usage", {})
       from .usage_meter import add_usage
       add_usage(u.get("prompt_tokens", 0), u.get("completion_tokens", 0), u.get("total_tokens", 0))
   except Exception:
       pass
   ```
3. **任务首尾收口**：`service/meeting/tasks.py` 的 `process_meeting`：
   - 开头 `start_metering()`；
   - 结尾（`finally` 前）`collect()` 拿总量，写一条 `task_usage`，并
     `UPDATE users SET tokens_used_this_month = tokens_used_this_month + :total`（注意跨月重置：
     若 `quota_period_start` 不是本月，先清零再累加）。
4. **新模型注册**：建 `models/usage.py`（`TaskUsage`），在 `models/__init__.py` 导出 →
   `create_all` 会自动建表。
5. **提交前校验**：`create_tasks` 里加"`tokens_used_this_month` ≥ `monthly_token_quota` 则 429"。
   > 注意：token 是**事后**才知道精确值，提交前只能按"上月均值 / 固定预估"做软拦截。
   > 策略建议：提交前只拦"已超额"的，不预扣；跑完后据实累加。能接受轻微超支。
6. **月度重置**：给 beat 加一个每日任务，检查并重置跨月用户的 `tokens_used_this_month`
   （或在累加时惰性重置，见第 3 步）。

**验收**：跑一个任务后 `task_usage` 有记录、`users.tokens_used_this_month` 增长；超额用户提交被拦。

> ✅ **已落地（2026-06-02）**：
> - `models/usage.py` `TaskUsage`（已在 `models/__init__.py` 注册，`create_all` 会自动建；迁移 SQL 也含）。
> - `service/quota.py` `record_task_usage(db, task_id, user_id, usage)`：写一条 `task_usage` + **行锁**(`with_for_update`)累加 `users.tokens_used_this_month`，累加前 `reset_if_new_month` 跨月惰性清零。
> - `tasks.py` `process_meeting`：在 `[TOKEN计量]` 日志后调 `record_task_usage`，失败 `rollback`+日志、不阻断主流程。
> - 提交前校验在 `create_tasks` 第 ③ 步：`tokens_used_this_month ≥ monthly_token_quota` 才 429（软拦，不预扣；token 是事后量）。≤0 配额=不限。
> - 月度重置策略：**惰性**（累加时 + 读取时都按 `quota_period_start` 是否本月判），未上 beat 定时重置任务（不需要）。

### 工作四：先压测摸产能（不写功能，拿数字）

**目标**：在堆功能前，实测**单 worker 能并发跑几个任务、瓶颈在哪（LLM 限流 / CPU / 内存）**，
用真实数字反推第 2.3 节星级配额该设多少。

**步骤**：

1. 造 3~5 个代表性样本（短文本、长文本、长音频各一）。
2. 单 worker 设 `-c 4`，同时投 8~10 个任务，观察：
   - 实际并行度（看 `running` 数）、单任务耗时、是否触发 LLM 429/超时；
   - worker 进程内存峰值（langgraph 吃内存）、CPU。
3. 逐步加 `-c`，找到"再加并发反而变慢 / 开始报限流"的拐点 = **单 worker 安全并发数**。
4. 查 DashScope 账号的 **QPS / 并发上限**（这通常是真正的天花板，不是你的机器）。
5. 产出结论填进本文 2.3：N 个 worker × 单 worker 并发 = 集群总产能；星级配额按此分配。

**运行要点（踩过的坑）**：
- **worker 用 prefork，别用 `run_worker.py`**：`celery -A core.celery_app.celery_app worker -l info -c N`。
  `run_worker.py` 是 `--pool=solo -B` 单机形态，solo 只 1 并发，测不出 `-c`。Windows 同理测不出（solo/threads）。
- **临时抬并发上限**：`meeting_router.py:42` `MAX_CONCURRENT_PER_USER=2` 会拦住批量提交，压测期间改大（如 50），工作一会用星级逻辑替换它。
- **第一条任务丢弃**：worker 启动预加载 langchain（`tasks.py` 顶部 import，冷启动 ~46s），首个任务偏慢，测 warm 的。
- **token 数从 `[TOKEN计量]` 日志读**；单任务时长从 DB `started_at`/`finished_at` 读（已自动记录）。
- **先纯文本后音频**：文本隔离 LLM 流水线信号，音频另测（多 ASR 耗时 + ASR 自身限流）。
- **测量只需 dev 运行时**（uvicorn + prefork worker + docker compose 的 pg/redis + 真实 DeepSeek key），**不用全套生产部署**。
- **环境坑（CentOS/Aliyun ECS）**：系统 python 常是 3.6.8，太老（langchain/langgraph 0.2 要 3.10+）。**别动系统 python**（yum 依赖它），用 miniconda 建 3.11 环境：`conda create -n meeting python=3.11 -c conda-forge --override-channels -y`（`-c conda-forge` 绕开 Anaconda TOS；建好若无 pip 再 `conda install -c conda-forge pip`）。

### 工作五：多机部署改造（横向扩 worker 的前置）

**目标**：单台 4C8G 不够时（突发 100 提交要 ~1.5–2h 排空，见 0.5），横向扩到 ~4 台 worker。
前提是先解除项目里"API 与 worker 同机、共用本地磁盘"的硬假设。

**核心问题链**（本地磁盘跨机必断）：
```
API 存音频 (meeting_router create_tasks 写本地盘)
 → worker 读音频跑 ASR (tasks.py 读 task.audio_path)              ← 跨机读不到！
 → worker 写 md/pdf/docx/transcript (tasks.py)
 → API 读文件下载 (download_task/batch_zip) + _to_out 的 os.path.exists 判 has_xxx  ← 跨机读不到！
```

**必做（阻塞项）**：

1. **共享存储**（最大工作量）：
   - **方案 A（推荐，几乎不改码）**：阿里云 NAS/NFS 挂所有机器同一路径，环境变量 `MEETING_DATA_ROOT`
     指过去（env override 已留好，`tasks.py:38`）。API 机与 worker 机该值必须一致。
   - **方案 B（更云原生，要改码）**：产物上 OSS（`.env` 已有 OSS 凭证，ASR 在用）。需改：
     写产物→上传 OSS、下载→预签名 URL、`os.path.exists`→OSS 存在性检查。工作量大于 NAS。
   - 涉及文件：`service/meeting/tasks.py`(写产物) / `router/meeting_router.py`(存音频/下载/has_xxx) / `service/meeting/cleanup.py`(删目录/音频)。
2. **改 worker 启动方式**（`run_worker.py` 现为单机 dev 形态，自己注释都写"单 worker 部署"）：
   - `--pool=solo` → Linux 用 prefork `-c 4~6`（solo 只 1 并发，跑不满 0.5 节算的 ~6 并发）
   - **去掉 `-B`（内嵌 beat）** → beat 独立成**全集群唯一**实例（每台机带 `-B` 会让清理任务每天触发 N 次）
   - 多机 beat 用 **RedBeat**（调度状态存 Redis，DEPLOYMENT.md 第 3 节）
   - `--without-gossip/mingle` 可保留：`delete_task` 的 `control.revoke(terminate=True)` 走 broker 广播，跨机仍生效，不依赖 gossip
3. **配置中心化**：
   - `REDIS_HOST` / `POSTGRES_HOST`：从 `localhost` 改指中心实例 + 放通安全组
   - **`JWT_SECRET_KEY` 所有 API 机必须一致**（HS256 对称密钥，否则 A 机签的 token 到 B 机验不过 → 随机掉登录）
   - `MEETING_DATA_ROOT` 所有机一致指向共享存储

**Redis / PG 放哪（决策）**：
- **单机 / 双机阶段**：Redis 直接 co-locate 在那台 4核8G 上（`docker compose up -d redis`）。本项目 Redis
  负载极轻——broker 只传 task_id 整数、结果也小，实占 ~100-200MB，不值得单独开机器；代价仅约 1 个并发任务的内存。
- **扩到 3-4 台 worker 时**：Redis 是共享 broker，所有 worker 都连它 → 挪到**阿里云托管 Redis(Tair)** 或
  一台 **1核2G 小机**，避免 ①单点故障（broker 挂则全队列停）②与 worker 抢内存。PG 已托管，Redis 同理。
- **PG**：已是阿里云托管 RDS（2G/3100 IOPS），所有机器 `POSTGRES_HOST` 指它即可，不要在 worker 机上再起 docker postgres。

**应做**：

4. API 多实例 + 负载均衡（SLB/nginx）；JWT 无状态，**不需要 sticky session**，随便轮询。
5. **数据库迁移单独跑一次**：现在每个 API 进程启动都 `create_all`（`app_main.py:24`），多机同启会
   竞争建表偶发报错，且加列不生效 → 部署阶段跑一次迁移（Alembic / 手动 SQL），别靠应用启动。
6. 集中日志 + NTP 时钟同步（beat 调度 / 任务 ETA 依赖各机时钟一致）。

**不用管**：Redis 应用层 cache/session（`redis_client.py`，已 Redis 后端，天然多机安全）；
LLMClient/config 单例（进程内状态，每 worker 进程一份，无跨机问题）。

**验收**：API 在 A 机、worker 在 B 机，提交**音频**任务能跑通全流程（A 存音频→B 读到→B 写产物→A 下载到）；
起 2 台 worker，删任务的 `control.revoke` 能终止任一台上正在跑的任务；全集群只有一个 beat 在触发清理。

## 5. 关键陷阱速查（踩过的 / 易踩的）

- **加列不生效**：`create_all` 不 ALTER 已有表，`users` 加列必须手动迁移（见 2.4）。新建**表**才会自动。
- **Redis 优先级方向**：0 高还是 9 高有歧义，**实测后再固定**（工作一第 3 步）。
- **beat 全集群唯一**：多个 beat 会重复触发定时任务（`DEPLOYMENT.md` 已警告），多机用 RedBeat。
- **.env 加载顺序**：`core/database.py` 顶部有防御性 `load_dotenv`，**别删**（`DEPLOYMENT.md` 有事故记录，celery worker 进程模块加载早于 celery_app 的 load_dotenv）。
- **LLMClient 是单例**（`minutes_engine.py:322`）：token 计量别想着改它的构造，挂在 `invoke_safe` 里用 contextvar 区分任务。
- **token 是事后量**：提交前只能软拦"已超额"，无法精确预扣。
- **连接池 × 进程数**：gunicorn -w 4 × pool 20 = 80 连接，别超 PG `max_connections`（默认 100）。
- **本地磁盘假设（多机阻塞）**：API 存音频 / worker 写产物 / API 读下载，全走本地盘 → 多机必须先上共享存储（NAS/OSS），详见工作五。
- **JWT 密钥多机要统一**：HS256 对称密钥，多 API 机 `JWT_SECRET_KEY` 不一致会随机掉登录。
- **worker 启动参数**：`run_worker.py` 的 `--pool=solo`+`-B` 是单机形态，多机要换 prefork 且 beat 独立单实例。
