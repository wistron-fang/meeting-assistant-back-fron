## 0. 速读（30 秒了解全貌）

- **它是什么**：把会议「文本/上传文档/音频」转成结构化会议纪要（Markdown + PDF + DOCX）的异步系统。
- **技术栈**：FastAPI(API) + Celery(异步 worker) + PostgreSQL(元数据) + Redis(消息队列/缓存) + 阿里云 DashScope(LLM) + 阿里云 NLS/OSS(语音识别)。
- **核心特征**：单任务是**重活长耗时**（ASR + LLM 7 步流水线，~15 分钟），所以一切异步化；用**星级**决定排队优先级和配额。
- **关键认知**：「100 用户在线」≠「100 任务并发」。HTTP 层很轻，真瓶颈是 LLM 限流 + worker 内存。详见第 9 节。
- **生产形态**：单台阿里云香港 ECS（4核8G），三个进程由 systemd 托管（开机自启 + 崩溃自愈）：`meeting-api`、`meeting-worker`、`redis`。PG 是托管 RDS，Redis 本机。

## 1. 架构

### 1.1 组件图

```
                ┌─────────────┐
   浏览器/前端 ──HTTP──▶│  meeting-api │  uvicorn app_main:app --workers 4  (:8100)
                │  (FastAPI)  │  - 鉴权(JWT) / 提交任务 / 查状态 / 下载产物
                └──────┬──────┘
                       │ apply_async(带 priority)         ┌──────────────┐
                       ▼                                   │  PostgreSQL   │
                ┌─────────────┐   读写元数据/配额  ◀───────▶│  (托管 RDS)   │
   Redis ◀─broker─┤ Celery 队列 │                          │ users /       │
   (本机:6379)   └──────┬──────┘                           │ meeting_tasks │
                       │ 取任务(高星先取)                   │ task_usage    │
                       ▼                                   └──────────────┘
                ┌─────────────────────────────────────┐
                │ meeting-worker (celery -c 32 -B)     │
                │  process_meeting 单任务流水线：       │
                │   1) (音频)ASR 转写  ← 阿里云 NLS/OSS │
                │   2) generate_minutes  ← DashScope LLM(deepseek) 7步, ~15min
                │   3) 渲染 PDF / DOCX                  │
                │   4) 写产物到 MEETING_DATA_ROOT 本地盘 │
                │  + 内嵌 beat：每日 cleanup_expired    │
                └─────────────────────────────────────┘
```

### 1.2 两条主流程

**A. 提交任务（HTTP，快）**
1. 前端带 JWT 调 `POST /meeting/...`（`router/meeting_router.py`）。
2. **三重配额拦截**（`service/quota.py`）：①并发数 ②本月任务数 ③本月 token，任一超限直接拒。
3. 写一行 `meeting_tasks`(status=pending)，`apply_async(priority=star_to_priority(用户星级))` 投进 Redis 队列，立即返回 task_id。

**B. 执行任务（Celery worker，慢，~15min）** —— `service/meeting/tasks.py::process_meeting`
1. worker 按**优先级**从队列取任务（高星=小数字=先取）。
2. 开一个 DB session（贯穿整个任务，但靠频繁 commit 把连接还池）。
3. 音频源 → 先 ASR 转写（`service/meeting/asr.py`，阿里云 NLS）回填 source_text。
4. `generate_minutes`（`service/meeting/minutes_engine.py`）跑 LLM 流水线，langgraph 编排，调 DashScope `deepseek-v4-pro`。进度靠 `StageCapture` 抓 stdout 写回 DB。
5. token 计量：`usage_meter.py` 用 contextvar 累加，任务结束 `collect()` → 写 `task_usage` + 累加 `users.tokens_used_this_month`。
6. 渲染 PDF(`pdf_renderer.py`)、DOCX(`docx_renderer.py`)，产物落 `MEETING_DATA_ROOT/<user_id>/<task_id>/`。
7. status→done。音频源文件在完成 5 分钟后由 `cleanup_audio` 延时任务删除。

## 3. 运行时拓扑（生产，单台 ECS）

| 进程 | 由谁托管 | 命令 | 职责 |
|---|---|---|---|
| **meeting-api** | systemd | `uvicorn app_main:app --host 0.0.0.0 --port 8100 --workers 4` | HTTP API（4 进程抗轮询） |
| **meeting-worker** | systemd | `celery -A core.celery_app.celery_app worker -l info -c 32 -B` | 跑任务（32 并发）+ 内嵌 beat 定时清理 |
| **redis** | systemd | redis-server | Celery broker/backend + 应用缓存 |
| **PostgreSQL** | 阿里云托管 RDS | — | 元数据/配额，`POSTGRES_HOST` 指过去 |

- 代码：`/opt/meeting/backend`，应用目录 `/opt/meeting/backend/app`，**`PYTHONPATH=/opt/meeting/backend/app`**（扁平 import 依赖它）。
- Python：miniconda 环境 `meeting`(py3.11)，路径 `/root/miniconda3/envs/meeting`。
- 数据产物：`MEETING_DATA_ROOT`（生产 = `/opt/meeting/data/meeting`），在 backend 目录**之外**。
- systemd unit 文件：`/etc/systemd/system/meeting-api.service`、`meeting-worker.service`（仓库副本在 `backend/`）。

> ⚠️ **API 与 worker 同机、共用本地磁盘**是当前硬假设（API 存音频→worker 读、worker 写产物→API 读下载）。要扩多机必须先上共享存储(NAS/OSS)，见 `生产上线与并发改造方案.md` 工作五。

## 4. 数据库 Schema（PostgreSQL）

> 建表方式：`app_main.py` 启动时 `Base.metadata.create_all`。**只会建新表，不会给已存在的表加列** —— 给 `users` 等加列必须手动 `ALTER TABLE`（项目虽装了 alembic 但未真正接入迁移）。

### users
| 列 | 类型 | 说明 |
|---|---|---|
| id | UUID(pk) | uuid4 |
| username / email | varchar uniq | |
| hashed_password | varchar | bcrypt |
| is_active / is_superuser | bool | |
| **star_level** | smallint, default **4** | 星级 1~5，驱动优先级 + 配额默认 |
| max_concurrent | smallint null | 并发覆盖；NULL=取星级默认 |
| monthly_task_quota | int null | 月任务数覆盖 |
| monthly_token_quota | bigint null | 月 token 覆盖；≤0=不限 |
| tokens_used_this_month | bigint default 0 | 本月已用（跨月惰性重置） |
| quota_period_start | date null | 本月起算日（每月1号） |

### meeting_tasks
`id(int pk)` / `user_id(fk users)` / `title` / `source_type(paste|upload|audio)` / `source_text` / `output_dir` / `status(pending|running|done|failed)` / `progress` / `celery_id` / `audio_path` / `transcript_path` / `md_path` / `pdf_path` / `docx_path` / `rag_json_path` / `stage` / `error` / `created_at` / `started_at` / `finished_at`。时间均为**北京时间 naive**。

### task_usage
单任务 token 审计：`task_id(fk, CASCADE)` / `user_id` / `model` / `prompt_tokens` / `completion_tokens` / `total_tokens` / `calls` / `created_at`。

### 星级默认配额（`service/quota.py::STAR_QUOTAS`，现行值）
| 星级 | 并发 | 月任务数 | 月 token | 队列优先级（**小=先跑**） |
|---|---|---|---|---|
| 5 | 5 | 200 | 18,000,000 | 1 |
| 4（默认） | 4 | **20** | 9,000,000 | 3 |
| 3 | 3 | 50 | 4,500,000 | 5 |
| 2 | 2 | 20 | 1,800,000 | 7 |
| 1 | 1 | 10 | 900,000 | 9 |

> ⚠️ **优先级方向已实测**（2026-06-02 ECS）：本套 Celery+Redis「数字越小=优先级越高」。改 priority 列务必保持 star5→1。

## 5. 配置参考（`backend/.env`）

`.env` 放在 `backend/.env`，被 `app_main.py` / `core/database.py` / `core/celery_app.py` 各自防御性 `load_dotenv`。**含密钥，绝不进 git。**

### 会议后端必需的键
| 键 | 用途 |
|---|---|
| `POSTGRES_HOST/PORT/USER/PASSWORD/DB` | PG 连接（生产指 RDS 内网地址） |
| `REDIS_HOST/PORT/PASSWORD` | Redis（本机一般 localhost:6379，无密码留空） |
| `JWT_SECRET_KEY` | ⭐JWT 对称密钥（HS256）。多 API 机必须一致 |
| `JWT_ALGORITHM` / `JWT_ACCESS_TOKEN_EXPIRE_MINUTES` | 默认 HS256 / token 有效期（约 14 天） |
| `DASHSCOPE_API_KEY` / `DASHSCOPE_BASE_URL` | ⭐LLM（阿里云 DashScope，deepseek 系列） |
| `LLM_MODEL` / `LLM_FALLBACK_MODEL` / `LLM_TEMPERATURE` / `OPENAI_MODEL` | 主/备模型与温度 |
| `ALIYUN_ACCESS_KEY_ID/SECRET` / `ALIYUN_NLS_APPKEY` | 语音识别（NLS） |
| `ALIYUN_OSS_BUCKET/ENDPOINT` | ASR 音频中转 OSS |
| `MEETING_DATA_ROOT` | 产物根目录（生产 `/opt/meeting/data/meeting`） |
| `TIMEZONE` / `APP_ENV` / `MEM_LIMIT` | 时区 / 环境标识 / 内存提示 |

### 可选 / 有代码默认值
| 键 | 默认 | 说明 |
|---|---|---|
| `DB_POOL_MODE` | `queue` | 设 `null` → 用 NullPool（worker 用） |
| `DB_POOL_SIZE` / `DB_MAX_OVERFLOW` / `DB_POOL_RECYCLE` | 5 / 10 / 1800 | 连接池（API 在 systemd 里覆盖为 3/3） |
| `MEETING_RETENTION_DAYS` | 7 | 清理保留期 |
| `ALLOW_REGISTRATION` | 关 | 自助注册开关；默认关，注册返 403 |

> `.env` 里还有一批 `ES_/MILVUS_/BID_/JUHE_/BOCHA_/SERPER_/OPENROUTER_/DOCMIND_` 等键——那是同仓库另一功能(kb_chat/FinInsRAG)的，**会议后端用不到**，新部署可不填。

### 连接池关键设计（PG `max_connections` 仅 70，预算紧）
- **API**：小 QueuePool，systemd 里内联 `DB_POOL_SIZE=3 DB_MAX_OVERFLOW=3`（4 进程 ×6=24 封顶，轮询实际只用 ~8）。
- **worker**：`DB_POOL_MODE=null`（NullPool，用完即关）。否则 `-c 32` prefork 每进程留 1 条 idle ×32 会吃光 70。
- ⚠️ 池参数**不能写进 `.env`**（API+worker 共用同一 .env 会互相污染），只能在 systemd unit 的 `Environment=` 或 start 脚本内联前缀里分别给。

## 6. 从零部署到一台同规格机器（step-by-step）

> 目标机：阿里云 ECS 4核8G（`ecs.e-c1m2.xlarge`），系统 Alibaba Cloud Linux / CentOS。前提：已有托管 PG(RDS) 和各云服务密钥。运维入口：阿里云控制台 **Workbench 网页终端**（无需本地 SSH/密钥）。

### 6.1 准备 Python 环境
```bash
# 别动系统自带 python（yum 依赖它）。装 miniconda，建 3.11 环境。
bash ~/miniconda.sh -b -p /root/miniconda3        # 若未装 miniconda 先下载
source /root/miniconda3/bin/activate
conda create -n meeting python=3.11 -c conda-forge --override-channels -y
conda activate meeting
```

### 6.2 放代码 + 装依赖
```bash
mkdir -p /opt/meeting
# 用 Workbench 上传 backend 代码包(tar.gz)，解压到 /opt/meeting/backend
tar -xzvf /root/backend_prod_xxx.tar.gz -C /opt/meeting   # 得到 /opt/meeting/backend
cd /opt/meeting/backend
# 用最小依赖子集，别装全量 requirements.txt（混了别的功能的重包）
pip install -i https://mirrors.aliyun.com/pypi/simple/ -r requirements-meeting.txt
```

### 6.3 配置 .env
```bash
# 把生产 .env 放到 backend/.env（从备份恢复，或照 .env.example 填）。
# 必填见第 5 节。务必确认：POSTGRES_*、REDIS_*、JWT_SECRET_KEY、DASHSCOPE_*、ALIYUN_*、MEETING_DATA_ROOT。
vi /opt/meeting/backend/.env
```

### 6.4 装 Redis（本机，systemd 自启）
```bash
yum install -y redis || dnf install -y redis
```

### 6.5 初始化数据库
```bash
source /root/miniconda3/bin/activate meeting

# create_all 建新表（users/meeting_tasks/task_usage）。无 psql 也能跑：
```

### 6.6 建第一个用户（自助注册默认关）
```bash

# 设为 superuser / 高星可直接在 RDS 改 users 表（app 实时读库，免重启）。
```

### 6.7 装 systemd 服务（开机自启 + 崩溃自愈）
仓库 `backend/meeting-api.service`、`backend/meeting-worker.service` 已写好（路径/环境变量都烤进去了）。要点：
- `ExecStart` 用 conda 绝对路径：`/root/miniconda3/envs/meeting/bin/{uvicorn,celery}`
- `WorkingDirectory=/opt/meeting/backend/app`、`Environment=PYTHONPATH=/opt/meeting/backend/app`
- API：`Environment=DB_POOL_SIZE=3` `DB_MAX_OVERFLOW=3`；Worker：`Environment=DB_POOL_MODE=null`
- 都 `Restart=always`

```bash
cp /opt/meeting/backend/meeting-api.service /opt/meeting/backend/meeting-worker.service /etc/systemd/system/
systemctl daemon-reload
systemctl enable --now meeting-api meeting-worker
```

## 8. 容量与并发约束（给配额/扩容决策用）

- **核心认知**：「N 用户在线」≠「N 任务并发」。HTTP 轻；单任务重（~15min，ASR+LLM）。
- **实测产能**（4核8G，2026-06-01）：单任务 ~71k token / 15.5min / 25 次 deepseek 调用；token 速率恒定 ~4,620/min。
- **瓶颈排序**：① embedding RPS（-c 越高 429 越多，是真上限）② DeepSeek TPM 120万/分 ③ worker 内存（langgraph 0.5~1G/任务）。**内存实测不是瓶颈**（COW 共享）。
- **结论**：单台 `-c 32` 压测 0 失败，足以撑「~100 在线用户」，**多机非必需**。峰值 ~20 在线用户时连接数 ~25-40，远低于 PG 70。
- **提速头号杠杆**：LLM chat 流水线并行化（15min→~5min）；但 **embedding 不能再并行**（已是瓶颈）。

## 9. 关键陷阱速查

1. **加列不生效**：`create_all` 不 ALTER 已有表，给 `users` 加列必须手动 `ALTER TABLE`。
2. **PYTHONPATH 必设**：worker/脚本不 `export PYTHONPATH=/opt/meeting/backend/app` 会 `No module named 'service'`。
3. **别 pkill 在跑的 worker**：会把在跑任务变僵尸（DB 卡 running，celery 默认不自动重跑）。优雅重启用 `systemctl restart`（发 SIGTERM 等任务）。清僵尸：`UPDATE meeting_tasks SET status='failed' WHERE status='running'`。
4. **连接池 × 进程数**：池参数别写 `.env`（API+worker 共用会污染）；worker 必须 `DB_POOL_MODE=null`，否则 32 进程 idle 连接吃光 PG 70。
5. **优先级方向**：本套 Redis「数字越小=优先级越高」，star5→priority 1。改 STAR_QUOTAS 别写反。
6. **依赖装最小子集**：用 `requirements-meeting.txt`，别装全量（混了 opencv/onnxruntime/pymilvus 等会议没 import 的重包）。
7. **`.env` 防御性加载别删**：`database.py`/`celery_app.py` 顶部的 `load_dotenv` 是 worker 进程早于入口加载时的兜底，删了会连错库。
8. **beat 单机唯一**：worker 的 `-B` 内嵌 beat 仅单机可用；扩多机 worker 必须去掉 `-B` 改独立 RedBeat，否则每台都触发重复清理。
9. **多机硬假设**：API/worker 共用本地磁盘写死了，跨机前必须先上共享存储（见改造方案工作五）。
10. **自助注册默认关**：`ALLOW_REGISTRATION` 不设=注册返 403；建号走 `admin_create_user.py`。
